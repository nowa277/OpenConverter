## 三、事故复盘

**`gh release download` 在 tmpfs 静默截断。** 把 v0.2.2 的 6 个桌面包（实为 68-131 MB）下载到 `/tmp`（tmpfs），全部被截断到 ~18 MB，但 `gh` 退出码 0、无报错。已基于坏文件算出 SHA256SUMS，差点把 6 个坏包签上 hash 当作 v1.0.0 推出去。最后一步用 GitHub API 的 `digest` 字段逐个比对真实 sha256 才发现。

**教训（已记入 ledger）：** release 资产必须对照 GitHub API digest 校验 sha256，不能信任下载工具的退出码；大文件暂存避免用 tmpfs，改用真磁盘目录。

## 四、下一步建议

按"分平台持续迭代"的既定方向，按优先级排列：

### 高优先级

1. **Android 真机验收（vivo Y78）** — UI redesign spec 里 M8.3 标注"UI 落地后做"。UI 已落地，应在真机上跑一遍 NCM/KWM/KGM/QMC → MP3/FLAC，重点验证 FGS 后台/锁屏转码、SAF 写入、自适应图标在国产 ROM 上的渲染。这是把 v1.0.0 从"能用"推向"可信"的关键一步。

2. **Android 生产签名** — 当前 APK 是 debug 签名，国产 ROM 会提示"未知开发者"。需生成 release keystore、配置 `build.gradle.kts` 的 `signingConfigs`，才能进正式分发渠道。keystore 密码不入库（用 `local.properties` 或环境变量）。

3. **QMCv2（.mflac / .mgg / .bkc）** — 三平台都还没实现，需 ekey，是当前最大的功能缺口。这是 v2 阶段的核心目标，三平台应同步设计 ekey 获取/传入方案。

### 中优先级

4. **OSS licenses 屏幕** — ffmpeg-kit (full-gpl) 捆绑大量 GPL 库，Android 端应有合规的许可证展示页（v1 显式 defer 了）。合规风险，建议在 Android 进 Play Store / 正式分发前补上。

5. **CI（GitHub Actions）** — 目前三平台构建全靠本地，无回归防护。建议加：JVM 测试（必跑）+ 桌面 `npm run build` + Android `:app:assembleDebug`，PR 触发。成本低、收益高。

6. **解密器 parity 双向回流** — Android Kotlin 解密器与 JS 解密器约定 1:1 等价（`expected-sha256.json` oracle）。任一端修 bug，另一端必须同步，否则 parity 测试会暴露但容易被忽略。建议把 parity 校验纳入 CI。

### 低优先级 / 待澄清

7. **版本号长期策略** — 记忆里曾记"三平台统一版本号"，但本次发布桌面 0.2.2 / Android 1.0.0 分开了（用户已确认接受）。需澄清：是长期独立维护，还是某个里程碑（如 QMCv2 落地）后重新统一。这影响后续 release 命名。

8. **桌面端跟进** — `windows-installer` 分支有 `convertOne` 路径冲突修复（2364a30），确认是否已充分测试、是否需要发 v0.2.3 桌面小版本。

### 仓库治理

9. **清理暂存盘** — `/home/user/oc-release-staging/`（~682 MB）、`/tmp/openconverter-backup-*`（filter-repo 前的备份）、`/tmp/oc-*` 临时目录，确认 v1.0.0 验收无误后可删。

10. **staging 脚本固化** — 本次下载/校验/发布流程靠临时脚本（`/home/user/oc-release-staging.sh`、`oc-wait-staging.sh`）。若计划持续发版，可整理成 `scripts/release-*.sh` 入库，避免下次重蹈截断覆辙。
